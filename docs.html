OpenSpec 完整架構文件

本文件以 @fission-ai/openspec v1.3.1 原始碼為基底, 詳細拆解 OpenSpec CLI 工具的入口、build、命令、工作流引擎、AI 工具 adapter 系統, 以及怎麼擴展或 fork 出自己的版本.

TypeScript Node.js ≥20.19 ESM Commander.js Inquirer Vitest Zod 28+ AI 工具

01專案是什麼

OpenSpec 是一套「規格驅動開發 (spec-driven development)」CLI 工具. 它的目標不是直接寫程式碼, 而是替 AI coding agent (Claude Code, Cursor, Codex, Gemini, 28+ 工具) 產生一致的 slash command 與 agent skill 檔案, 把開發流程切成可追蹤的 artifact: proposal → specs → design → tasks → apply → archive.

輸入

使用者在 AI 工具裡輸入 /opsx:propose "加深色模式"

中介

OpenSpec 預先產生的 skill / command 引導 AI 建立 spec, design, tasks 檔案

輸出

可驗證的 openspec/changes/<name>/ 目錄結構, 並能 archive 回到 specs

OpenSpec 把「變更」(change) 與「事實規格」(spec) 切開: change 描述要怎麼從現況走到目標, archive 後合併回 spec

02技術 Stack

純 Node.js CLI, 沒有資料庫, 沒有伺服器. 全部都靠 npm package + 檔案系統運作.

Runtime 與語言

ESM 模組, package.json "type": "module"
TypeScript 5.9.3
target ES2022, NodeNext, strict 模式
pnpm v9

05入口點 (Entry Points)

OpenSpec 有 三個入口, 分別對應不同使用情境:

① CLI

npm i -g @fission-ai/openspec 後在 PATH 取得 openspec:

npm bin
  → bin/openspec.js  (package.json#bin)
       → dist/cli/index.js  (Commander 註冊)

② Library

作為 Node 套件 import (用得不多, 但已暴露):

import * as openspec from '@fission-ai/openspec';
// 內含 cli + core
// 主要是 getGlobalConfig() 之類

③ Nix Flake

NixOS 使用者可透過 flake 安裝:

nix run github:Fission-AI/OpenSpec
# 由 flake.nix 定義, CI 有跑驗證

CLI 入口檔逐行

// package.json
{
  "name": "@fission-ai/openspec",
  "type": "module",
  "bin": { "openspec": "./bin/openspec.js" },
  "exports": { ".": { "types": "./dist/index.d.ts", "default": "./dist/index.js" } },
  "files": ["dist", "bin", "schemas", "scripts/postinstall.js"]
}

package.json 三個關鍵欄位: bin 註冊 CLI, exports 給 library, files 控制 npm 打包內容

// bin/openspec.js  ← 整個檔案就這兩行
#!/usr/bin/env node
import '../dist/cli/index.js';
為什麼這樣設計?
  • bin shim 永遠是 JS, 不會被 TS 編譯; 避免 build 後 shebang 遺失.
  • 真正的程式碼放在 dist/cli/index.js, 由 tsc 從 src/cli/index.ts 產生.
  • scripts/postinstall.js 在 npm install 後執行, 提示使用者 opt-in shell completion.

execFileSync(process.execPath, [tsc, '--version'], { stdio: 'inherit' });

09init 完整流程

這是整套工具最有料的命令, 也是 InitCommand (約 780 行) 做的事情. 拆成 9 個 phase:

sequenceDiagram autonumber actor U as 使用者 participant CLI as cli/index.ts participant IC as InitCommand participant FS as FileSystem participant CG as command-generation participant TPL as templates U->>CLI: openspec init CLI->>IC: new InitCommand(opts) IC->>FS: 1. validate() 檢查目錄 + 寫入權限 IC->>FS: 2. detectLegacyArtifacts() 偵測舊版檔案 IC->>U: 互動: 確認清理舊檔? IC->>FS: 3. cleanupLegacyArtifacts() 移除舊 marker IC->>FS: 4. getAvailableTools() 掃描 .claude/, .cursor/, ... IC->>FS: 5. migrateIfNeeded() 既有專案升級 IC->>U: 6. searchableMultiSelect 選工具 IC->>FS: 7. createDirectoryStructure() 建 openspec/ 與子目錄 loop 每個選到的工具 IC->>TPL: getSkillTemplates(workflows) IC->>TPL: getCommandContents(workflows) IC->>CG: CommandAdapterRegistry.get(toolId) IC->>CG: generateCommands(contents, adapter) CG-->>IC: GeneratedCommand[] IC->>FS: 8. writeFile() 寫到 .{tool}/skills/ + 對應路徑 end IC->>FS: 9. createConfig() 寫 openspec/config.yaml IC->>U: 顯示成功訊息 + 提示重啟 IDE

關鍵設計點

Profile × Delivery 矩陣

Profile (要哪些 workflow) 來自 global-config; Delivery (skills/commands/both) 也是. 兩者組合決定 init 真正寫出哪些檔.

Extend mode

若 openspec/ 已存在, init 走 extend mode: 不顯示 welcome screen, 不建 config (除非沒有), 只刷新 skill/command.

非互動模式

沒 TTY 或 --tools 帶值時走 non-interactive. 沒指定且偵測不到任何工具會直接拋錯, 不會默默裝空的.

刪除舊檔

切換 delivery 時自動清掉另一邊的檔案: 如從 both → commands, 會刪掉 .{tool}/skills/openspec-* 目錄.

10工作流引擎 (Workflow Engine)

OpenSpec 真正的核心抽象是「workflow schema」, 定義在 schemas/spec-driven/schema.yaml, 也允許專案 fork 出自訂 schema 到 openspec/schemas/<name>/.

schema.yaml 結構

name: spec-driven
version: 1
description: Default OpenSpec workflow - proposal → specs → design → tasks
artifacts:
  - id: proposal
    generates: proposal.md
    template: proposal.md
    instruction: |
      Create the proposal document that establishes WHY...
    requires: []        # 沒有依賴, 第一個產生

  - id: specs
    generates: "specs/**/*.md"
    template: spec.md
    instruction: |
      Create specification files...
    requires: [proposal]

  - id: design
    generates: design.md
    template: design.md
    instruction: |
      Create the design document...
    requires: [proposal]

  - id: tasks
    generates: tasks.md
    template: tasks.md
    instruction: |
      Create the task list...
    requires: [specs, design]   # 必須等兩者都做完

apply:
  requires: [tasks]
  tracks: tasks.md
  instruction: |

13AI 工具 Adapter 系統

OpenSpec 一次支援 28 個 AI 工具. 機制是「tool-agnostic 內容 + per-tool 格式化 adapter」.

核心抽象

// src/core/command-generation/types.ts
export interface CommandContent {
  id: string;          // 'explore'
  name: string;        // 'OpenSpec Explore'
  description: string;
  category: string;
  tags: string[];
  body: string;        // markdown 指令本體
}

export interface ToolCommandAdapter {
  toolId: string;
  getFilePath(commandId: string): string;       // 例如 .claude/commands/opsx/explore.md
  formatFile(content: CommandContent): string;  // 加上 frontmatter
}

三個 adapter 範例

檔案位置: .claude/commands/opsx/<id>.md · YAML frontmatter: name, description, category, tags

export const claudeAdapter: ToolCommandAdapter = {
  toolId: 'claude',
  getFilePath(id) {
    return path.join('.claude', 'commands', 'opsx', `${id}.md`);
  },
  formatFile(c) {
    return `---
name: ${escapeYamlValue(c.name)}
description: ${escapeYamlValue(c.description)}
category: ${escapeYamlValue(c.category)}
tags: ${formatTagsArray(c.tags)}
---

${c.body}
`;
  },
};

檔案位置: .cursor/commands/<id>.md · Cursor 用 mdc 格式, 與 Claude 略有差異

結構相同, 但 frontmatter 與路徑會根據 .cursor/ 規範調整. 看 src/core/command-generation/adapters/cursor.ts.

特殊處: 全域 scope. Codex 不寫到專案內, 寫到使用者全域 prompt 目錄, 因此 getFilePath 回傳「絕對路徑」.

InitCommand 看到 path.isAbsolute() 為 true 時不會再 prepend 專案根目錄.

14Profile / Delivery 系統

同樣是「分離關注點」的設計: Profile 決定要哪些 workflow, Delivery 決定怎麼交付.

Profile

  • core 預設 4 個 workflow: propose, explore, apply, archive
  • custom 由 config.json 內的 workflows: [] 自選

完整可用 workflow: propose, explore, new, continue, apply, ff, sync, archive, bulk-archive, verify, onboard

Delivery

  • both 同時產生 skill + slash command (預設)
  • skills 只產生 agent skill
  • commands 只產生 slash command

15如何擴展功能

三種最常見的擴展需求, 每種都有對應的「最小修改集合」.

A. 新增一個 CLI 命令

  1. 在 src/commands/ 建立 my-cmd.ts, 匯出一個 class 或 function
  2. 在 src/cli/index.ts 註冊 program.command('my-cmd')
  3. 把純邏輯放到 src/core/ (方便單測)
  4. 在 test/commands/ 寫 vitest 測試
  5. 跑 pnpm build && node bin/openspec.js my-cmd 驗證
範例: 新增 openspec hello 
// src/commands/hello.ts
export class HelloCommand {
  async execute(name = 'world') {
    console.log(`Hello, ${name}!`);
  }
}

// src/cli/index.ts (新增)
import { HelloCommand } from '../commands/hello.js';
program
  .command('hello [name]')
  .description('Say hello')
  .action(async (name?: string) => {
    await new HelloCommand().execute(name);
  });

B. 新增一個 AI 工具 Adapter

  1. 在 src/core/config.ts#AI_TOOLS 加一行: { name, value, skillsDir, available: true }
  2. 在 src/core/command-generation/adapters/<tool>.ts 實作 ToolCommandAdapter
  3. 在 src/core/command-generation/adapters/factory.ts + registry.ts 註冊
  4. 若該工具的 skill 目錄結構特別, 在 src/core/init.ts 的特殊處理區看是否需加例外
  5. 跑 pnpm test; 補 fixture 與 snapshot

C. 新增一個 Workflow Schema

例如想加入 review 階段 (tasks 完成後要人 review 才能 apply):

  1. 跑 openspec schema fork my-schema, 會複製預設 schema 到 openspec/schemas/my-schema/
  2. 編輯 schema.yaml, 加新 artifact: 
      - id: review
    generates: review.md
    template: review.md
    instruction: |
      Document review findings...
    requires: [tasks]
  3. 在 openspec/schemas/my-schema/templates/ 加 review.md template
  4. 修改 apply.requires 改成 [review]
  5. 在 openspec/config.yaml 把 schema: 改成 my-schema
  6. 跑 openspec schemas 確認被列出; openspec instructions review --change xxx 確認可產生

D. 新增一個 Workflow (skill + slash command)

  1. 在 src/core/templates/workflows/ 建 my-workflow.ts, export getMyWorkflowSkillTemplate() + getOpsxMyCommandTemplate()
  2. 在 src/core/templates/skill-templates.ts re-export
  3. 在 src/core/profiles.ts#ALL_WORKFLOWS 加入 id
  4. 在 src/core/shared/ 的 getSkillTemplates() / getCommandContents() 對應加入
  5. (可選) 在 WORKFLOW_TO_SKILL_DIR 映射加入
  6. 跑 openspec update 驗證它出現在 .claude/skills/

16客製化 / Fork 指南

如果想做出「自己版本的 OpenSpec」, 建議按這個順序走:

flowchart TB S[1. 評估保留 vs 改寫範圍] --> A[2. fork repo + 重新命名 package] A --> B[3. 確認 schema 是否要客製] B --> C[4. 客製 workflow templates] C --> D[5. 調整 AI 工具支援清單] D --> E[6. 改 init 預設 profile / delivery] E --> F[7. 改 brand: welcome screen, palette, README] F --> G[8. 重設 changesets + CI + npm scope] G --> H[9. 自家 dogfood: 在 openspec/ 跑 init]

建議保留

穩定且通用

  • artifact-graph + schema 載入
  • parsers (markdown / requirement-blocks)
  • validation 三層架構
  • Commander 註冊風格
  • command adapter pattern
  • 跨平台 utils

較容易改

  • src/core/config.ts#AI_TOOLS 工具清單
  • src/core/templates/workflows/* 每個 workflow 內容
  • src/ui/welcome-screen.ts brand
  • src/core/styles/palette.ts 配色
  • schemas/spec-driven/ 預設 schema
  • src/core/profiles.ts 預設 workflow 組合

逐步操作

第 1 步: package.json 重新命名 
{
  "name": "@your-scope/your-cli",
  "version": "0.1.0",
  "bin": { "your-cli": "./bin/your-cli.js" },
  ...
}

記得改 bin/openspec.js 檔名 + 對應 import path. 也要改 GitHub repo URL.

第 2 步: 客製 workflow templates

每個 workflow 的 SKILL.md 內容都在 src/core/templates/workflows/:

// 範例: src/core/templates/workflows/explore.ts
export function getExploreSkillTemplate() {
  return {
    dirName: 'openspec-explore',
    name: 'openspec-explore',
    description: '...',
    body: `# Explore workflow ...`,  // ← 改這裡
  };
}

改完後 pnpm build, 在測試專案跑 openspec update --force 就會刷新.

第 3 步: 客製 schema 與 templates

把 schemas/spec-driven/ 複製成 schemas/my-spec/, 修改後在 init 預設指向新 schema:

// src/core/init.ts
const DEFAULT_SCHEMA = 'my-spec';   // ← 改這裡

// src/commands/workflow/shared.ts
export const DEFAULT_SCHEMA = 'my-spec';
第 4 步: 限縮 AI 工具清單

多數團隊只用 1-3 個工具. 把 src/core/config.ts#AI_TOOLS 裡不需要的 entry 拿掉 / 設 available: false, 同時刪掉對應 adapter 檔. InitCommand 會自動只顯示剩下的選項.

第 5 步: 改預設 profile 與 delivery 
// src/core/global-config.ts
const DEFAULT_CONFIG: GlobalConfig = {
  profile: 'core',     // 或 'custom'
  delivery: 'both',    // 改成 'skills' 或 'commands'
  workflows: [],       // 若 profile='custom' 想預填
  featureFlags: {},
};
第 6 步: 改 brand
  • src/ui/welcome-screen.ts ASCII art
  • src/ui/ascii-patterns.ts 圖樣
  • src/core/styles/palette.ts chalk 顏色
  • 輸出訊息搜尋 "OpenSpec" 字串
第 7 步: 重新打通 release pipeline
  1. 刪掉 .changeset/ 既有 entry, 留 config.json
  2. 把 .github/workflows/release-prepare.yml 內 secret/scope 改成自家
  3. 第一次發布: pnpm changeset → commit → CI 跑 release
實戰建議: 不要一次改全部. 先 fork → 改 brand → 跑通 init → 在實際專案 dogfood 兩週 → 再開始改 schema / workflow. 這個工具的價值來自一致性, 改太多反而失去重點.

18測試策略

使用 Vitest, 設定在 vitest.config.ts, 有 vitest.setup.ts 全域 hook.

測試目錄組織

test/
├── cli-e2e/            # 跑真 CLI binary, 行為驗證
│   └── basic.test.ts
├── commands/           # 每個 command class 一個 spec
├── core/               # 純邏輯單測 (parsers, validator, graph...)
├── prompts/            # 自訂 multi-select 元件
├── utils/              # 跨平台 util
├── fixtures/           # 範例 markdown / config
├── helpers/            # tmpdir, mock fs 等
└── specs/              # 對 schema 本身的 sanity test

測試金字塔

單元 (core/*)

最多, 純函式; 不碰 FS

整合 (commands/*)

用 tmpdir, 跑完整 command class

E2E (cli-e2e/*)

spawn 真 binary, 驗 stdout/exit code

跨平台原則 (來自 openspec/config.yaml)

  • 所有路徑用 path.join() / path.resolve(), 永遠不寫死 /
  • 測試的 expected 路徑也要 path.join() 組
  • 檔案大小寫敏感性要考慮
  • 動到路徑邏輯時, 新增「Windows 路徑處理 scenario」
